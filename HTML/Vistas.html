3.2 Vistas Arquitectónicas

DECISIONES ARQUITECTONICAS

1: Seguridad en el acceso a la BD

Contexto: El sistema manejará datos sensibles de clientes, proveedores e inventario, por lo que se requiere proteger la información contra accesos no autorizados.

Decisión:Se definió que solo la capa de negocio tendrá acceso directo a la base de datos (MySQL), utilizando credenciales encriptadas y evitando el acceso desde la capa de presentación.

Consecuencias:

  • Positivo: Aumenta la seguridad, evita fugas de información.
  • Negativo: Puede dificultar pruebas rápidas desde el cliente si no hay acceso directo a BD.

2: Autenticación y manejo de usuarios

Contexto: Se identificó la necesidad de diferenciar roles (administrador, mecánico, cliente) para limitar accesos según el perfil.

Decisión: Implementar un sistema de autenticación con roles y permisos, donde el administrador gestiona usuarios y privilegios.

Consecuencias:

  • Positivo: Previene que un usuario sin permisos acceda a funciones críticas (ej. eliminar inventario).
  • Negativo: Aumenta la complejidad de desarrollo y mantenimiento de roles.

3: Rendimiento en consultas

Contexto: El sistema debe consultar inventario y diagnósticos en tiempo real, incluso con grandes volúmenes de datos.

Decisión: Optimizar la base de datos con índices en campos de búsqueda frecuentes (ejemplo: código de repuesto, placa de la moto, fecha de diagnóstico).

Consecuencias:

  • Positivo: Mejora la velocidad de búsqueda y reduce tiempos de respuesta.
  • Negativo: Mayor espacio en disco y mantenimiento de índices al modificar registros.

4: Escalabilidad con microservicio de IA

Contexto:Se requiere un módulo de IA de diagnóstico que podría crecer de forma independiente en el futuro.

Decisión: Desplegar la IA como un microservicio independiente que se comunica vía API con el sistema principal.

Consecuencias:

  • Positivo: Escalable, se puede mejorar la IA sin afectar el resto del sistema.
  • Negativo: Incrementa la complejidad de despliegue y requiere asegurar la comunicación entre servicios.

5: Mantenibilidad con arquitectura por capas

Contexto:Se busca que el sistema sea fácil de mantener y extender con nuevas funciones (reportes, nuevos tipos de diagnósticos).

Decisión: Adoptar una arquitectura por capas (presentación, negocio, datos) separando responsabilidades y facilitando pruebas unitarias.

Consecuencias:

  • Positivo: Facilita mantenimiento, reduce impacto de cambios.
  • Negativo: Puede aumentar la cantidad de código inicial y la curva de aprendizaje.

DESCRIPCIONES ARQUITECTONICAS

Incidentes

Esta imagen nos muestra como incidentes se encarga de registrar, actualizar y consultar los eventos relacionados con emergencias o situaciones médicas reportadas. Permitiendo así asociar datos clave como síntomas, lugar de ocurrencia, paciente involucrado y atención recibida, garantizando trazabilidad en todo el proceso.

Clasificación de incidentes

Nos muestra como se clasifica y organiza los distintos tipos de incidentes que pueden ocurrir, facilitando la identificación, categorización y análisis posterior de patrones o estadísticas de emergencias.

Sintomas

Permite registrar y consultar los síntomas reportados por los pacientes o identificados por el personal de salud, funcionando como base para diagnósticos y predicciones.

Categoría de síntomas

La imagen muestra cómo se organiza y categoriza los diferentes síntomas posibles, asegurando que la información registrada sea uniforme y estandarizada para análisis médicos posteriores.

Acciones del personal de salud

Nos muestra cómo se gestiona las acciones realizadas por el personal de salud frente a un incidente o consulta, incluyendo primeros auxilios, tratamientos aplicados y derivaciones a centros médicos.

Modalidades de atención

Se define y clasifica las modalidades de atención médica prestada, tales como atención en sitio, hospitalaria o remota, facilitando la estandarización del proceso asistencial.

Pacientes

El diagrama nos muestra cómo se administra la información personal y médica de cada individuo atendido en el sistema, incluyendo historial clínico, antecedentes y contactos de emergencia.

Clasificación de usuarios

Se puede observar la clasificación de los usuarios del sistema según criterios médicos o administrativos (ej. adultos, niños, afiliados, no afiliados), optimizando la gestión de casos y estadísticas.

Valoraciones clínicas

Permite registrar las primeras valoraciones clínicas realizadas por el personal de salud o el sistema antes de un diagnóstico completo, agilizando la atención inicial.

Resultados clínicos

Administra los resultados clínicos finales basados en los síntomas, antecedentes y pruebas médicas, aportando información clave para la atención y el seguimiento del paciente.

Personal de salud

Se puede observar cómo se gestiona la información de los trabajadores del sistema de salud (médicos, enfermeros, técnicos), incluyendo roles, turnos y asignaciones.

Perfiles del personal

La imagen muestra cómo se clasifica los diferentes perfiles dentro del personal de salud, facilitando la organización de responsabilidades y la correcta asignación de funciones.

Medios de traslado

Administra los medios de traslado utilizados para movilizar pacientes o recursos médicos en caso de incidentes, incluyendo disponibilidad, rutas y tiempos de respuesta.

Clasificación de transporte

Se puede visualizar como se categoriza los distintos medios de transporte (ambulancia, helicóptero, vehículo particular), garantizando un registro organizado y útil para la logística de emergencias.

Centros médicos

Centraliza la información de los centros médicos disponibles, su capacidad, ubicación y servicios prestados, permitiendo referencias rápidas y coordinadas durante una emergencia.

Clasificación de instituciones

Clasifica las instituciones médicas según su nivel de atención (básico, especializado, de alta complejidad), facilitando la asignación correcta de pacientes.

Datos históricos

Se encarga de almacenar de forma sistemática todos los datos ingresados en el sistema, desde incidentes hasta diagnósticos, garantizando trazabilidad y acceso a la información histórica.

Gestión de accesos

Se gestiona el acceso y seguimiento de incidentes, sitios de atención y registros de sucesos, permitiendo búsquedas rápidas y eficientes por parte de usuarios y personal de salud.

Reportes

Se genera reportes consolidados sobre síntomas frecuentes, incidentes recurrentes y estadísticas de sucesos, aportando información útil para la toma de decisiones y prevención.

Privilegios de usuarios

Administra las configuraciones y privilegios asociados a cada tipo de perfil de usuario, garantizando que cada rol solo acceda a las funciones que le corresponden.

Gestión de cuentas

Gestiona las cuentas de acceso de los distintos usuarios del sistema, controlando autenticación, roles y permisos para proteger la información sensible.

Nodos identificados

    1. Cliente:

  • Aplicación web en el PC de escritorio.
  • Se conecta al servidor mediante protocolo HTTPS.
  • Requiere navegador web actualizado.

    • 2. Servidor de Aplicaciones :

  • Implementado en Java.
  • Contiene la lógica de negocio (gestión de incidentes, síntomas, pacientes, registros, etc.).
  • Expone servicios REST/HTTP para que el cliente consuma.

    • 3. Base de Datos :

  • MySQL como gestor de datos.
  • Contiene todas las tablas que ya tienes modeladas (Incidente, Paciente, Personal, Registro, Diagnóstico, etc.).
  • El servidor de aplicaciones se conecta a la BD a través de un driver JDBC.

    • 4. Servicio IA (Opcional) :

  • Microservicio externo que se puede integrar (ej. para clasificación automática de síntomas o predicción de diagnósticos).
  • Puede ejecutarse en un contenedor independiente (Docker/Kubernetes).
  • El servidor de aplicaciones envía datos y recibe la respuesta procesada.

    • Explicación de Conexiones y Restricciones :

  • Cliente → Servidor de Aplicaciones
    • Comunicación vía HTTPS para garantizar seguridad.
    • Autenticación de usuarios (roles: administrador, personal de salud, usuario).
  • Servidor de Aplicaciones → Base de Datos
    • Conexión mediante JDBC/MySQL.
    • Restricción: la conexión debe estar protegida con usuario/contraseña cifrados y se recomienda firewall para que solo el servidor acceda a la BD.
    • Integridad de datos garantizada mediante transacciones.
  • Servidor de Aplicaciones → Servicio IA
    • Comunicación vía API REST o gRPC.
    • Restricción: uso de tokens de autenticación para consumir el microservicio.
    • Opcional, por lo que el sistema debe seguir funcionando, aunque no esté disponible.
  • Base de Datos
    • Restringir acceso directo de los clientes finales (nunca deben conectarse a la BD).
    • Copias de seguridad periódicas.
    • Control de concurrencia para múltiples usuarios.
