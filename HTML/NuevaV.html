DECISIONES ARQUITECTONICAS

Estrategia de persistencia de datos para el sistema de primeros auxilios.

Contexto: El sistema requiere almacenar y gestionar información sensible y crítica (pacientes, incidentes, síntomas, diagnósticos, registros médicos, personal de salud). Se necesita garantizar integridad, seguridad, disponibilidad y recuperación de los datos. Además, el sistema debe soportar múltiples usuarios concurrentes y permitir escalabilidad futura.

Decisión:Se decidió utilizar MySQL como motor de base de datos relacional, gestionado desde el Servidor de Aplicaciones en Java mediante DAO/ORM (Hibernate/JPA). La estrategia incluye:

  • Definición de claves primarias, foráneas e índices para integridad y rendimiento.
  • Cifrado de contraseñas y datos sensibles.
  • Transacciones ACID para garantizar consistencia en concurrencia.
  • Backups automáticos (incrementales diarios, completos semanales).
  • Replicación para alta disponibilidad.
  • Uso de roles y permisos para control de acceso.
  • Persistencia de archivos (reportes, documentos) en almacenamiento externo, guardando solo la ruta en la base de datos.

Consecuencias:

  • Positivo:
    • Seguridad reforzada (cifrado, HTTPS, roles).
    • Integridad de datos asegurada con restricciones y transacciones.
    • Disponibilidad y tolerancia a fallos mediante replicación y respaldos.
    • Escalabilidad para soportar mayor carga de usuarios.
    • Separación clara entre lógica de negocio y persistencia con DAO/ORM.
  • Negativo:
    • Mayor complejidad en la configuración y administración de la BD (replicación, backups).
    • Costos adicionales de almacenamiento para respaldos y archivos externos.
    • Dependencia tecnológica en MySQL (migrar a otro motor puede requerir esfuerzo adicional).

Estrategia de Autenticación con Doble Token y Código de Verificación

Contexto: El sistema gestiona información médica y personal crítica, por lo que es necesario reforzar la autenticación. La autenticación tradicional con usuario y contraseña no es suficiente, ya que puede ser vulnerable a robo de credenciales. Se busca una solución que combine tokens seguros para sesiones y un código de verificación dinámico como segundo factor de autenticación.

Decisión: Se decidió implementar un flujo de autenticación de doble token con MFA:

  • 1.Ingreso inicial:
    • El usuario proporciona su usuario y contraseña.
    • Si son válidos, el sistema genera un código de verificación (OTP) y lo envía al usuario (correo electrónico, SMS o app autenticadora).

  • 2.Verificación del código:
    • El usuario ingresa el código recibido.
    • Si el código es correcto y no ha expirado, se genera el Access Token (vida corta, ej. 15 min) y el Refresh Token (vida más larga, ej. 7 días).

  • 3.Acceso a recursos:
    • El cliente usa el Access Token para consumir APIs seguras.
    • o Al expirar, el cliente usa el Refresh Token para solicitar un nuevo Access Token, sin reingresar usuario/contraseña, pero el Refresh Token puede invalidarse si se detecta anomalía.

Consecuencias:

  • Positivo:
    • Máxima seguridad: incluso si roban la contraseña, necesitan el código de verificación.
    • Access Token expira rápido, reduciendo impacto de un posible robo.
    • El Refresh Token permite experiencia fluida sin pedir credenciales cada vez.
    • Compatible con estándares modernos de seguridad (MFA + JWT).
  • Negativo:
    • Incrementa la complejidad técnica (gestión de códigos OTP y doble token).
    • Requiere infraestructura adicional (servicio de correo/SMS o integración con apps autenticadoras).
    • Puede afectar la experiencia del usuario si no se diseña bien (ej. tiempos de espera para recibir el código).
    • Costos adicionales si se usan servicios de envío de SMS.

Mecanismo de manejo de errores y excepciones en el sistema de primeros auxilios.

Contexto: El sistema interactúa con varios componentes (cliente web, servidor de aplicaciones en Java, base de datos MySQL, servicio IA opcional). Durante la operación pueden presentarse fallos como:

  • Errores de conexión a la base de datos.
  • Fallos en servicios externos (ej. IA, notificaciones).
  • Datos inválidos enviados por los usuarios.
  • Excepciones no controladas en el servidor.
Es necesario un mecanismo que garantice estabilidad, trazabilidad y una buena experiencia de usuario, evitando caídas totales del sistema.

Decisión: Se definió una estrategia de manejo de errores centralizado, que incluye:

  • 1.En el Servidor de Aplicaciones (Java):
    • Uso de try/catch globales y manejadores centralizados de excepciones (ej. @ControllerAdvice en Spring Boot).
    • Clasificación de errores en:
      • Errores de cliente (400 – Bad Request, 401 – Unauthorized, 403 – Forbidden, 404 – Not Found).
      • Errores de servidor (500 – Internal Server Error, 503 – Service Unavailable).

  • 2.En la Base de Datos:
    • Manejo de transacciones con rollback automático en fallos.
    • Validación de integridad antes de insertar/actualizar registros.

  • 3.En el Cliente (Web/Móvil):
    • Mensajes claros y comprensibles para el usuario (sin detalles técnicos).
    • Reintentos automáticos en fallos temporales (ej. pérdida de conexión).

  • 4.Registro y Monitoreo:
    • Logs estructurados (ej. Log4j, SLF4J) con nivel de severidad (INFO, WARN, ERROR, FATAL).
    • Alertas automáticas para errores críticos.
    • Integración futura con herramientas de monitoreo (ej. ELK Stack, Grafana).

Consecuencias:

  • Positivo:
    • Sistema más estable y tolerante a fallos.
    • Experiencia de usuario mejorada (mensajes claros, no técnicos).
    • Facilita la detección y solución rápida de errores mediante logs.
    • Evita pérdidas de datos al usar rollback en transacciones fallidas.
    • Mejora la seguridad al no exponer detalles técnicos al cliente.
  • Negativo:
    • Complejidad adicional en la configuración del manejador centralizado de errores.
    • Posible sobrecarga en almacenamiento de logs si no se gestionan con rotación.
    • Requiere disciplina del equipo para registrar y categorizar errores correctamente.

Mecanismo para el consumo de APIs externas en el sistema de primeros auxilios

Contexto: El sistema puede requerir integrarse con servicios externos para enriquecer su funcionalidad, como:

  • Servicios de geolocalización (ej. Google Maps, OpenStreetMap) para ubicar al paciente.
  • Servicios de mensajería (SMS, correo, notificaciones push).
  • Servicios de IA externos o APIs de salud.
Dado que estos servicios son de terceros, es necesario definir cómo consumirlos de manera segura, eficiente y tolerante a fallos.

Decisión: Se decidió implementar una capa de integración para APIs externas con las siguientes características:

  • Acceso seguro:
    • Uso de HTTPS para todas las comunicaciones.
    • Manejo de tokens de autenticación (API Keys, OAuth2) almacenados de forma segura en el servidor.
  • Abstracción en el Servidor de Aplicaciones (Java):
    • Implementar clientes REST dedicados (ej. con RestTemplate o WebClient en Spring Boot).
    • Uso de interfaces y adaptadores para desacoplar la lógica del negocio del proveedor específico.
  • Manejo de errores y tolerancia a fallos:
    • Reintentos automáticos en fallos temporales.
    • Circuit Breaker para evitar saturar al sistema en caso de caída del servicio externo (ej. con Resilience4j).
    • Mensajes claros al usuario cuando un servicio externo no esté disponible.
  • Control y monitoreo:
    • Logs dedicados para registrar cada interacción con APIs externas.
    • Métricas de rendimiento (tiempo de respuesta, disponibilidad).
  • Flexibilidad futura:
    • Los servicios externos deben ser fácilmente sustituibles (ej. cambiar de proveedor de geolocalización sin modificar la lógica del negocio).

Consecuencias:

  • Positivos:
    • Integración sencilla con múltiples proveedores externos.
    • Seguridad reforzada con tokens y HTTPS.
    • Alta resiliencia gracias a estrategias de reintentos y circuit breakers.
    • Escalabilidad y flexibilidad al desacoplar la lógica del negocio de las dependencias externas.
  • Negativos:
    • Dependencia de servicios de terceros (riesgo si cambian políticas, costos o disponibilidad).
    • Incremento en la complejidad del sistema (módulos de integración, manejo de tokens, fallos externos).
    • Posibles costos adicionales por uso de APIs externas (ej. SMS o mapas de Google).

Estrategia para cambio de tecnología o migración en el sistema de primeros auxilios

Contexto: Con el crecimiento del sistema y su adopción, puede surgir la necesidad de cambiar de tecnología o migrar por motivos como:

  • Escalabilidad (más usuarios y datos de sensores).
  • Limitaciones de la tecnología actual (ej. MySQL vs. bases de datos distribuidas).
  • Costos de licencias o infraestructura.
  • Integración con nuevas plataformas (móvil, nube, IA).
  • Fin de soporte de versiones o librerías críticas.
Un cambio tecnológico sin planificación puede generar pérdida de datos, caídas del sistema o problemas de compatibilidad.

Decisión: Se decidió adoptar una estrategia controlada de migración tecnológica, con las siguientes pautas:

  • Evaluación previa:
    • Análisis de riesgos, costos y beneficios antes de migrar.
    • Comparación de alternativas (ej. MySQL → PostgreSQL, servidor on-premise → nube).
  • Diseño desacoplado:
    • Arquitectura basada en capas y APIs para minimizar dependencias.
    • Uso de ORM (Hibernate/JPA) en la capa de persistencia para facilitar cambio de motor de BD.
  • Migración de datos:
    • Estrategia de ETL (Extract, Transform, Load) para trasladar información sin pérdida.
    • Validación de integridad y consistencia tras la migración.
  • Migración progresiva:
    • Uso de entornos paralelos (ej. correr MySQL y PostgreSQL al mismo tiempo mientras se valida).
    • Pruebas de rendimiento y seguridad en el nuevo entorno antes del cambio definitivo.
  • Documentación y capacitación:
    • Registrar los cambios tecnológicos.
    • Capacitar al equipo en la nueva tecnología para evitar errores de operación.

Consecuencias:

  • Positivos:
    • Flexibilidad para evolucionar con nuevas tecnologías.
    • Mayor vida útil del sistema al no quedar atado a tecnologías obsoletas.
    • Mejor rendimiento y escalabilidad al adoptar plataformas más modernas.
    • Reduce riesgos de interrupción al usar migración progresiva y pruebas previas.
  • Negativos:
    • Costos adicionales (infraestructura, tiempo de migración, capacitación).
    • Complejidad técnica al mantener sistemas en paralelo durante la transición.
    • Riesgo de incompatibilidades si no se hace un análisis profundo antes del cambio.
    • Posible resistencia al cambio por parte del equipo.

Mecanismo de cambio de idiomas y búsqueda en el sistema móvil/web desarrollado en Flutter y Dart

Contexto: El sistema de primeros auxilios está siendo desarrollado en Flutter con Dart como lenguaje principal. Se requiere soportar múltiples idiomas en la interfaz de usuario y permitir búsquedas rápidas en los datos de pacientes, incidentes y síntomas.
El objetivo es garantizar accesibilidad y usabilidad tanto en el frontend como en la integración con la base de datos vía APIs.

Decisión:

  • Cambio de Idiomas (i18n en Flutter):
    • Se utilizará el paquete oficial flutter_localizations junto con intl.
    • Los textos estarán en archivos .arb (Application Resource Bundle), lo que permite manejar traducciones sin modificar el código fuente.
    • El usuario podrá seleccionar manualmente el idioma o se detectará automáticamente desde la configuración del dispositivo.
    • Idiomas iniciales: Español (es) e Inglés (en), con posibilidad de añadir más (ej. Portugués).
  • Búsqueda en la Aplicación:
    • Búsqueda local en Flutter: con ListView + SearchDelegate para filtrar listas de datos descargados (ej. pacientes en memoria).
    • Búsqueda remota vía API: el cliente enviará parámetros al servidor (ej. nombre, cédula, fecha de incidente), y el servidor devolverá resultados desde MySQL.
    • Se usarán índices en MySQL para optimizar búsquedas y filtros avanzados en el backend.
    • En caso de escalar, se evaluará integrar ElasticSearch o soluciones similares.

Consecuencias:

  • Positivos:
    • La aplicación será multi-idioma y accesible para usuarios de diferentes regiones.
    • Separación clara entre lógica de negocio y traducciones, facilitando mantenimiento.
    • Búsqueda eficiente tanto en datos locales como en base de datos remota.
    • Escalabilidad futura gracias al uso de APIs para búsquedas complejas.
  • Negativos:
    • Mayor complejidad en la gestión de archivos de traducción .arb.
    • Necesidad de mantener sincronizadas las traducciones al agregar nuevas funcionalidades.
    • Posible latencia en búsquedas remotas si la API o la red son lentas.
    • Mayor carga de pruebas para verificar consistencia entre idiomas.

Mecanismo para generación y exportación de reportes (Excel, PDF y otros formatos)

Contexto: El sistema de primeros auxilios debe permitir consultar, exportar y compartir reportes con información de:

  • Pacientes, síntomas e incidentes registrados.
  • Signos vitales recolectados desde los sensores.
  • Estadísticas y métricas (ej. frecuencia de incidentes, diagnósticos más comunes).
Estos reportes son necesarios para:
  • Toma de decisiones por personal médico y administradores.
  • Comunicación con familiares de pacientes.
  • Auditorías y procesos legales.
Por lo tanto, se necesita un mecanismo flexible que soporte diferentes formatos de salida (PDF, Excel, CSV, etc.).

Decisión:

  • En el Backend (Java + MySQL):
    • Generación de reportes mediante librerías especializadas:
      • Apache POI → para Excel.
      • iText o JasperReports → para PDF.
      • Exportación a CSV/JSON → para integraciones con otros sistemas.
    • Exposición de reportes mediante endpoints REST (ej. /reportes/incidentes?formato=pdf).
  • En el Frontend (Flutter/Dart):
    • Descarga y visualización de reportes desde la API.
    • Uso de librerías como:
      • pdf y printing → para generar/mostrar PDFs en el móvil.
      • excel → para exportar datos en hojas de cálculo locales.
    • Los reportes podrán:
      • Guardarse en el dispositivo.
      • Compartirse por correo, WhatsApp u otras apps.
  • Seguridad y Acceso:
    • Solo usuarios autorizados (según rol) podrán generar o acceder a ciertos reportes.
    • Los reportes con datos sensibles se protegerán con contraseñas o cifrado en el caso de PDF.

Consecuencias:

  • Impactos positivos:
    • Flexibilidad al soportar múltiples formatos (PDF, Excel, CSV).
    • Compatibilidad con diferentes usuarios (administradores, médicos, auditores).
    • Integración sencilla con otras herramientas externas (ej. Excel, Google Sheets).
    • Mejora en la comunicación de información crítica.
  • Impactos negativos:
    • Aumento de la complejidad técnica (gestión de librerías diferentes por formato).
    • Posible mayor consumo de recursos en el servidor al generar reportes grandes.
    • Necesidad de mantener consistencia entre reportes generados en backend y frontend.
    • Riesgo de filtración de datos sensibles si no se asegura la descarga y el almacenamiento.

Mecanismo de soporte y escalabilidad para el crecimiento de usuarios en el sistema de primeros auxilios

Contexto: El sistema puede iniciar con una base pequeña de usuarios (100), pero se proyecta un crecimiento acelerado (100,000+ en una semana). Este incremento exponencial impacta en:

  • Carga del servidor de aplicaciones (Java).
  • Consultas y transacciones en la base de datos (MySQL).
  • Consumo de APIs externas (geolocalización, notificaciones).
  • Capacidad de red y almacenamiento.
Se requiere una estrategia que garantice rendimiento, disponibilidad y escalabilidad sin afectar la experiencia del usuario.

Decisión:

  • Arquitectura escalable:
    • Uso de balanceadores de carga (Load Balancer) para distribuir peticiones entre múltiples instancias del servidor de aplicaciones.
    • Implementación de contenedores (Docker + Kubernetes) para desplegar servicios de forma ágil y escalable.
  • Base de datos (MySQL):
    • Replicación maestro-esclavo para dividir carga de lectura/escritura.
    • Particionado o sharding en caso de crecimiento extremo.
    • Uso de caché (Redis/Memcached) para reducir consultas repetitivas.
  • Backend (Java):
    • Optimización de endpoints y uso de asíncronía en operaciones pesadas.
    • Escalado horizontal (más instancias del servidor) y vertical (mejor hardware si es necesario).
  • Frontend (Flutter/Dart):
    • Implementar paginación e infinite scroll en búsquedas/listados.
    • Reducir llamadas innecesarias a la API con caché local.
  • Infraestructura:
    • Despliegue en la nube (ej. AWS, GCP, Azure) con escalado automático (Auto Scaling).
    • Monitoreo de métricas (CPU, memoria, latencia, tiempo de respuesta).

Consecuencias:

  • Impactos positivos:
    • El sistema podrá manejar un crecimiento masivo de usuarios sin caídas.
    • Escalabilidad automática gracias a la nube y contenedores.
    • Mejor tiempo de respuesta al usar caché y balanceo de carga.
    • Flexibilidad para añadir nuevas funcionalidades sin comprometer rendimiento.
  • Impactos negativos:
    • Costos de infraestructura aumentarán a medida que crece el número de usuarios.
    • Mayor complejidad en la administración del sistema (requiere DevOps/SRE).
    • Riesgo de cuellos de botella en la base de datos si no se monitorea y optimiza.
    • Dependencia de servicios en la nube (riesgo de costos imprevistos).

Mecanismo de integración con redes sociales en el sistema de primeros auxilios

Contexto: El sistema puede beneficiarse de la integración con redes sociales en distintos escenarios:

  • Autenticación rápida de usuarios (ej. login con Google, Facebook, Apple ID).
  • Notificaciones y comunicación con familiares o grupos de apoyo (ej. enviar alertas de emergencia por WhatsApp, Telegram, Messenger).
  • Difusión de información de salud y prevención en canales oficiales (ej. página de Facebook o Twitter/X del sistema).
Se requiere definir una estrategia segura y flexible para consumir APIs de redes sociales.

Decisión:

  • Autenticación Social (Social Login):
    • Integrar proveedores de identidad externos (Google, Facebook, Apple).
    • Uso de OAuth2 / OpenID Connect para obtener permisos y validar la identidad del usuario.
    • Sincronizar usuarios de redes sociales con la base de datos interna del sistema.
  • Comunicación y Notificaciones:
    • Envío de alertas a contactos de apoyo a través de APIs de mensajería (ej. WhatsApp Business API, Telegram Bot API, Twilio).
    • Configuración de mensajes automáticos cuando se detecte un incidente de salud.
  • Difusión de información (marketing/prevención):
    • Posibilidad de conectar el sistema con redes sociales oficiales para publicar mensajes preventivos o comunicados.
    • Uso de programación de publicaciones desde el servidor (ej. usando APIs de Facebook Graph o Twitter API).
  • Seguridad:
    • Almacenar de forma segura tokens de acceso de redes sociales.
    • Solicitar únicamente los permisos mínimos necesarios.
    • Cumplir con las políticas de privacidad de cada red social (ej. GDPR, Meta policies).

Consecuencias:

  • Impactos positivos:
    • Mejor experiencia de usuario con autenticación rápida mediante redes sociales.
    • Comunicación en tiempo real con familiares a través de canales conocidos (WhatsApp, Messenger).
    • Mayor alcance y visibilidad del proyecto con difusión en redes sociales.
    • Flexibilidad para integrar distintos proveedores gracias a OAuth2.
  • Impactos negativos:
    • Dependencia de proveedores externos (si cambian políticas o precios, el sistema debe adaptarse).
    • Complejidad adicional en la gestión de permisos y tokens.
    • Posibles costos por uso de APIs (ej. WhatsApp Business API).
    • Riesgo de exposición de datos si no se configuran correctamente los permisos y la seguridad.

Mecanismo de cumplimiento de regulaciones de protección de datos en el sistema de primeros auxilios

Contexto: El sistema almacena y procesa información altamente sensible:

  • Datos personales (nombre, cédula, teléfono).
  • Información médica (signos vitales, diagnósticos, incidentes).
  • Ubicación geográfica en tiempo real (GPS).
Por tanto, está sujeto a regulaciones de protección de datos como:
  • GDPR (Unión Europea).
  • HIPAA (Estados Unidos, para datos médicos).
  • Ley 1581 de 2012 (Colombia, protección de datos personales).
  • Normativas locales según país/región de despliegue.
El incumplimiento podría generar sanciones legales, pérdida de confianza y riesgos de seguridad.

Decisión: Se decidió implementar una estrategia de cumplimiento legal y técnico, que incluye:

  • Privacidad desde el diseño (Privacy by Design):
    • Recolectar solo los datos estrictamente necesarios.
    • Minimizar la exposición de datos personales en reportes y APIs.
  • Consentimiento informado:
    • Solicitar aceptación explícita del usuario antes de recolectar o compartir datos.
    • Incluir una política de privacidad clara dentro de la app.
  • Protección técnica de datos:
    • Cifrado en tránsito (HTTPS/TLS) y en reposo (cifrado en BD).
    • Hash seguro para contraseñas (bcrypt/Argon2).
    • Roles y permisos para restringir el acceso según perfil (administrador, médico, usuario).
  • Derechos del usuario:
    • Opción de consultar, corregir o eliminar sus datos.
    • Posibilidad de retirar consentimiento en cualquier momento.
  • Gestión de incidentes:
    • Procedimiento para notificar al usuario y a autoridades competentes en caso de brechas de seguridad.
  • Almacenamiento y retención:
    • Definir un tiempo máximo de retención de datos según normativas.
    • Eliminar o anonimizar datos cuando ya no sean necesarios.

Consecuencias:

  • Impactos positivos:
    • Cumplimiento legal con normativas internacionales y locales.
    • Mayor confianza de los usuarios en el sistema.
    • Reducción de riesgos legales y sanciones.
    • Seguridad reforzada al aplicar cifrado y control de accesos.
  • Impactos negativos:
    • Mayor complejidad en el desarrollo (implementación de políticas de privacidad, módulos de consentimiento).
    • Costos adicionales en auditorías, certificaciones o asesoría legal.
    • Restricciones en el uso de datos para análisis avanzados si el usuario no otorga consentimiento.

Mecanismo de integración y fusión de aplicaciones en el sistema de primeros auxilios

Contexto: El sistema podría evolucionar hacia la integración o fusión con otras aplicaciones o servicios, por ejemplo:

  • Una aplicación de emergencias médicas ya existente.
  • Una app de seguimiento de pacientes en hospitales.
  • Plataformas de bienestar y salud preventiva.
La fusión puede ser necesaria por:
  • Sinergia tecnológica (evitar duplicación de funciones).
  • Expansión de usuarios (consolidar comunidades o clientes).
  • Optimización de costos (un solo sistema en lugar de varios).
Sin una estrategia clara, la fusión podría generar problemas de duplicación de datos, incompatibilidades tecnológicas y pérdida de trazabilidad.

Decisión: Se decidió adoptar una estrategia de fusión progresiva y basada en interoperabilidad, con los siguientes lineamientos:

  • Arquitectura modular y basada en APIs:
    • Todas las aplicaciones involucradas deben exponer funcionalidades clave mediante APIs REST/GraphQL.
    • Se usará un bus de integración (API Gateway o ESB) para unificar accesos.
  • Unificación de datos:
    • Definición de un modelo de datos común para pacientes, incidentes y registros.
    • Uso de ETL (Extract, Transform, Load) para migrar o sincronizar bases de datos.
    • Evitar duplicación de información mediante identificadores únicos.
  • Experiencia de usuario:
    • Mantener una interfaz unificada en Flutter/Dart para los usuarios finales.
    • Posibilidad de integrar aplicaciones fusionadas en una super-app o con un sistema de Single Sign-On (SSO).
  • Seguridad y cumplimiento:
    • Tokens compartidos (OAuth2, JWT) para acceso entre aplicaciones fusionadas.
    • Cumplimiento con regulaciones de datos (GDPR, HIPAA, Ley 1581).
  • Fusión progresiva:
    • Etapa 1: Integración vía APIs manteniendo aplicaciones separadas.
    • Etapa 2: Migración gradual de datos y servicios comunes.
    • Etapa 3: Consolidación en una única plataforma.

Consecuencias:

  • Impactos positivos:
    • Mayor alcance y adopción al unificar usuarios y servicios.
    • Reducción de costos al evitar duplicación de esfuerzos.
    • Escalabilidad al diseñar con arquitectura modular y basada en APIs.
    • Posibilidad de interoperar con sistemas hospitalarios o de emergencias existentes.
  • Impactos negativos:
    • Complejidad en la integración y migración de datos.
    • Riesgo de incompatibilidades tecnológicas (bases de datos diferentes, lenguajes distintos).
    • Potencial resistencia de usuarios al cambio de plataforma.
    • Incremento en costos iniciales por desarrollo de APIs, migración y pruebas.

Mecanismo de inclusión y accesibilidad para personas con discapacidades en el sistema de primeros auxilios

Contexto: El sistema será utilizado por pacientes, familiares y personal de salud. Es fundamental garantizar que personas con discapacidad visual, auditiva o motora puedan interactuar con la aplicación sin barreras. Además, la OMS y normativas como WCAG (Web Content Accessibility Guidelines) obligan a que los sistemas de salud sean accesibles para todos los usuarios.

Decisión: Se decidió implementar una estrategia de accesibilidad integral en la aplicación, con las siguientes medidas:

  • Usuarios con discapacidad visual (ciegos o baja visión):
    • Uso de lectores de pantalla en Flutter (Semantics, screenReader).
    • Contrastes altos y modo oscuro para mejorar visibilidad.
    • Posibilidad de control por voz para navegar en la app.
  • Usuarios con discapacidad auditiva (sordos o hipoacúsicos):
    • Incorporación de subtítulos y transcripción automática en mensajes de voz o video.
    • Alertas visuales (iconos, vibración) en lugar de solo notificaciones sonoras.
    • Integración futura con lenguaje de señas digital (video o avatar).
  • Usuarios sordomudos:
    • Uso de texto y símbolos gráficos como alternativa principal.
    • Opción de comunicación mediante chat en tiempo real con contactos de apoyo.
  • Usuarios con discapacidad motora:
    • Botones grandes y bien espaciados.
    • Navegación simplificada (gestos mínimos).
    • Soporte para dispositivos de asistencia externa (ej. teclados especiales).
  • Normativas y estándares:
    • Cumplir con WCAG 2.1 AA para accesibilidad.
    • Pruebas con usuarios reales para validar la experiencia inclusiva.

Consecuencias:

  • Impactos positivos:
    • Mayor accesibilidad para todos los usuarios, sin importar sus limitaciones.
    • Cumplimiento con normativas internacionales y locales.
    • Incremento de confianza y reputación del sistema como herramienta inclusiva.
    • Posibilidad de expandir el uso de la app a un público más amplio.
  • Impactos negativos:
    • Mayor esfuerzo de diseño y desarrollo (UI/UX adaptada y pruebas de accesibilidad).
    • Aumento en los tiempos de desarrollo por necesidad de validaciones adicionales.
    • Posibles costos extra al integrar tecnologías como reconocimiento de voz o subtitulado automático.

Mecanismo para soportar múltiples tipos de dispositivos (móvil, tablet, computador) con interfaz adaptable

Contexto: El sistema de primeros auxilios debe ser accesible desde diferentes dispositivos:

  • Celulares (uso principal, app en Flutter).
  • Tablets (útiles en hospitales o personal médico).
  • Computadores de escritorio (para administración y monitoreo).
Sin un diseño adaptable, la interfaz puede romperse o ser poco usable en pantallas con diferentes tamaños y resoluciones.

Decisión: Se decidió implementar una estrategia de diseño responsive y multiplataforma:

  1. Framework principal:
    • Uso de Flutter/Dart, que permite desplegar en Android, iOS, Web y Desktop desde una sola base de código.
  2. Ajuste de interfaz según dispositivo:
    • Uso de MediaQuery y LayoutBuilder para adaptar tamaños de fuentes, botones y componentes.
    • Diseño escalable con widgets flexibles (Flexible, Expanded, Wrap, GridView).
    • Soporte para orientación vertical y horizontal.
  3. Soporte específico:
    • Móviles: interfaz optimizada para uso con una sola mano, notificaciones push.
    • Tablets: pantallas divididas (split view) para mostrar más información en paralelo.
    • Escritorio/Web: menús laterales, tablas más grandes, acceso rápido a reportes.
  4. Pruebas y QA:
    • Simuladores/emuladores de diferentes resoluciones.
    • Pruebas en dispositivos físicos clave (Android, iPhone, tablet, navegador web).

Consecuencias:

  • Impactos positivos:
    • Accesibilidad en múltiples dispositivos con una sola base de código.
    • Mejor experiencia de usuario sin importar el tamaño de pantalla.
    • Reducción de costos de desarrollo al no crear apps separadas para cada plataforma.
    • Escalabilidad futura: fácil expansión a web/desktop si se requiere.
  • Impactos negativos:
    • Mayor complejidad en el diseño UI (hay que pensar en varios tamaños de pantalla).
    • Algunas funciones nativas pueden requerir adaptaciones específicas por plataforma.
    • Riesgo de inconsistencias visuales si no se hacen pruebas en todos los dispositivos.

Mecanismo de integración de Realidad Aumentada en el sistema de primeros auxilios

Contexto: El sistema busca no solo detectar emergencias, sino también asistir al usuario y al personal médico/familiar. La Realidad Aumentada puede mejorar la experiencia en:

  • Capacitación y entrenamiento en primeros auxilios (ej. guías interactivas para RCP o uso de botiquines).
  • Visualización de signos vitales en tiempo real sobre un avatar o figura.
  • Geolocalización AR para identificar la ubicación exacta de un paciente en un entorno físico.
Sin embargo, esta funcionalidad no es crítica para el MVP inicial, sino un valor agregado en fases avanzadas.

  2. Aplicaciones prácticas iniciales:
    • Tutoriales interactivos de primeros auxilios: mostrar cómo aplicar compresiones torácicas con un modelo 3D.
    • Simulación de emergencias para entrenamiento de estudiantes o personal de salud.
    • Marcadores AR para guiar a familiares en la ubicación del paciente dentro de un lugar.
  3. Despliegue progresivo:
    • Fase 1: AR en capacitación y entrenamiento.
    • Fase 2: AR en visualización de datos biométricos.
    • Fase 3: AR en geolocalización avanzada y simulaciones de rescate.

Consecuencias:

  • Impactos positivos:
    • Diferenciación del sistema frente a la competencia.
    • Mejor experiencia de usuario con interfaces intuitivas y visuales.
    • Utilidad en educación médica, simulaciones y entrenamiento.
    • Posible interés de hospitales, universidades y aseguradoras en integrarlo.
  • Impactos negativos:
    • Mayor complejidad de desarrollo e integración (ARKit/ARCore).
    • Requiere dispositivos compatibles (no todos los celulares soportan AR).
    • Aumento en consumo de batería y recursos del dispositivo.
    • Funcionalidad secundaria: no es crítica para la detección y alerta de emergencias.
